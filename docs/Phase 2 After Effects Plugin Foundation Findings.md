# Phase 2 After Effects Plugin Foundation Findings

The Adobe SDK guide confirms that the official **Skeleton** sample remains the recommended starting point for effect development, while the SDK itself spans multiple plugin categories such as Effects, AEGPs, Artisans, and AEIOs. For this project, the strongest primary foundation is an **Effect plug-in** for host-native use inside After Effects, potentially complemented by an **AEGP or panel-style companion component** only if workflow needs exceed what effect controls can comfortably expose. The same documentation also notes that for modern panel-like UI, Adobe prefers a web-based panel approach rather than the old C++ panel approach, which matters if the final product needs a richer queue, diagnostics, or download manager interface.

The `mobile-bungalow/after_effects_cmake` repository is useful because it operationalizes Adobe's Skeleton sample into a more CI-friendly **CMake-based cross-platform build**. It demonstrates generation of a `.plugin` bundle on macOS via Xcode and a `.aex` on Windows via Visual Studio/MSBuild, making it a practical boilerplate candidate for an open-source repository. However, it is still only a thin adaptation of Skeleton, so it should be treated as a **build-system reference** rather than a complete product foundation.

At this point, the likely recommendation is to base the project on the **official AE SDK Skeleton effect sample**, modernized with **CMake** and repository conventions inspired by existing community templates. This should provide a more trustworthy long-term foundation than adopting an older or more weakly maintained abstraction layer.

Adobe's official installer guidance provides concrete distribution rules that should be reflected in the PRD. If cross-host compatibility is desired, plug-ins should generally install to the **common Adobe MediaCore plug-ins folder**; however, an AE-specific install location is still valid when the plugin depends on AE-only functionality. On Windows, these paths are discoverable through Adobe registry entries. On macOS, the common path is `/Library/Application Support/Adobe/Common/Plug-ins/7.0/MediaCore/`, while the AE-specific path is `/Applications/Adobe After Effects [version]/Plug-ins/`. This means the project should plan for installer scripts or packaged archives that support both the common path and an AE-only fallback.

Adobe's Apple Silicon guidance is equally important. The plugin should be shipped as a **macOS Universal binary** so the same bundle supports both Intel and Apple Silicon hosts. Adobe explicitly calls out the need to add `CodeMacARM64 {"EffectMain"}` alongside the Intel entry point in the PiPL resource, and notes that debugging on macOS 15+ requires an additional signing step. The guide also warns against allowing C++ exceptions to cross the `extern "C"` plugin boundary on Apple Silicon, which should become an explicit engineering constraint in the PRD.

These findings strengthen the case for a build pipeline that produces: a Windows `.aex`, a macOS universal `.plugin` bundle, and predictable installer/archive outputs that can be published as downloadable release assets in an open-source repository.
